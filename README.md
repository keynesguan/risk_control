## 一、产品定位

### 1.1 解决的问题

| 痛点 | 现状 | 本产品方案 |
| --- | --- | --- |
| 风险发现滞后且误杀高 | 传统决策树依赖相关性，遇大促等场景大面积误杀 | 因果决策树剔除虚假相关，算法出树 → LLM 出策略，每日自动挖掘因果特征 |
| 案件排查效率低 | 跨系统拉数据、手工拼报告，耗时长 | 输入 User_ID 即可一键生成研判报告与处置建议 |
| 方案沉淀困难 | 方案散落在文档、聊天记录中，无法复用 | LLM-Wiki 结构化知识网 + Git 版本控制，方案自动沉淀、持续迭代 |
| Text-to-SQL 不可控 | 大模型直连数据库，存在安全与准确性风险 | 预设 API 组合路由彻底替代 Text-to-SQL |
| 隐私泄露风险 | 明文数据直接进入大模型上下文 | 复合 DLP 网关（Schema + NLP/NER），PII 在出口前即被高精度清洗 |

### 1.2 目标用户

- **风控策略分析师**：日常风险巡检、策略制定与调优
- **案件研判人员**：可疑用户排查、处置决策
- **业务安全架构师**：新业务场景风控方案设计
- **风控团队负责人**：策略审批、效果度量

### 1.3 核心价值主张

**"发现 → 研判 → 方案"三位一体**，让风控团队从重复劳动中释放，聚焦高价值的判断与决策。

## 二、系统总架构

系统采用**总控路由 + 三智能体**分治架构，由顶层 Router 根据用户意图分发至对应智能体链路。

```
                    ┌─────────────────────────┐
                    │  智能风控助手 — 总控路由    │
                    │       (Router)           │
                    └────┬─────────┬─────────┬─┘
                         │         │         │
              ┌──────────┘         │         └──────────┐
              ▼                    ▼                    ▼
    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │  风险挖掘智能体   │  │  案件研判智能体   │  │ 风控解决方案智能体 │
    └─────────────────┘  └─────────────────┘  └─────────────────┘
```

### 路由策略

总控路由接收用户输入后，基于意图识别将请求分发至对应智能体链路：

| 意图信号 | 路由目标 |
| --- | --- |
| 涉及风险趋势、异常指标、策略挖掘 | 风险挖掘智能体 |
| 包含具体 User_ID、案件编号、排查请求 | 案件研判智能体 |
| 描述业务需求、方案设计、场景咨询 | 风控解决方案智能体 |

#### 多意图解析与任务编排

实际业务输入往往包含多个意图（如"帮我查一下用户 U_123 为什么被封，另外基于这个人的作弊手法生成一个新的风控方案"同时命中案件研判 + 解决方案）。Router 须具备多意图处理能力：

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1. 意图拆解 | 将混合 Prompt 拆分为独立子任务 | 如上例拆为"研判 U_123"+"生成防控方案" |
| 2. 依赖分析 | 识别子任务间的数据依赖关系 | "方案生成"依赖"案件研判"的结论输出 |
| 3. 有序编排 | 按依赖顺序串行或并行调度 | 先调用案件研判智能体输出结论，再将结论作为上下文注入解决方案智能体 |
| 4. 前端展示 | 向用户展示"多步执行计划" | 明确告知用户"将分两步执行：① 研判案件 → ② 生成方案"，避免黑盒等待 |

- 若子任务间**无依赖**（如同时查两个不同用户的案件），Router 可并行调度以缩短响应时间
- 若子任务间**有依赖**，上游智能体的输出自动作为下游智能体的上下文输入，用户无需手动复制粘贴

## 三、智能体一：风险挖掘

> 从数据中自动发现风险，生成管控策略，经三段式验证漏斗（历史推演 → 线上空跑 → 真实结算）后归档上线。

### 3.2 关键环节说明

#### 因果决策树：算法出树 → LLM 出策略

##### 为什么不用传统决策树

传统风控决策树（XGBoost / CART）依靠**相关性**而非**因果性**。例如发现"凌晨 3 点下单"与"欺诈"高度相关，就会生成拦截规则——但如果当天正好是双 11 大促，相关性规则会导致大面积误杀。

**因果决策树（Causal Decision Trees / Uplift Modeling）** 先用因果算法剔除"虚假相关"（如大促效应），找出真正导致风险的**因果特征**，再将其转化为人类可读的规则。这是业界前沿且高度可行的方案（学术界和头部大厂均有成熟实践）。

##### 分层架构

采用"**算法管算、大模型管翻译**"的严格分层，大模型不参与任何数学计算：

```
┌──────────────────────────────────────────────────────────┐
│ 输入层                                                    │
│ 上传观测数据：[用户特征, 干预动作(拦截/放行), 最终标签(黑/白)] │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│ 因果树生成层（传统算法侧，非大模型）                         │
│ 调用因果森林算法库（EconML / CausalML）                     │
│ 目标函数：最大化 Uplift（真实干预收益）                      │
│ 输出：因果决策树 JSON                                      │
│   - 每个节点：拆分特征 + 阈值 + 因果效应权重                 │
│   - 叶节点：对应的 Treatment Effect（干预增益）              │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│ LLM 策略转化层（大模型侧）                                  │
│ 输入：因果决策树 JSON                                      │
│ 输出：                                                    │
│   1. 自然语言风险归因报告（因果路径解释、业务影响分析）        │
│   2. 规则 DSL（可直接下发规则引擎的结构化表达）               │
│   3. 管控策略建议（含阈值调整方向与预估效果）                 │
└──────────────────────────────────────────────────────────┘
```

**因果决策树 JSON 示例：**

```json
{
  "tree_id": "CT_20260413_001",
  "target": "maximize_uplift",
  "nodes": [
    {
      "id": 0,
      "type": "split",
      "feature": "device_change_freq",
      "threshold": 2,
      "causal_effect": 0.73,
      "left": 1,
      "right": 2
    },
    {
      "id": 1,
      "type": "leaf",
      "causal_effect": 0.12,
      "sample_size": 8500
    },
    {
      "id": 2,
      "type": "split",
      "feature": "transaction_amount",
      "threshold": 1000,
      "causal_effect": 0.85,
      "left": 3,
      "right": 4
    },
    {
      "id": 3,
      "type": "leaf",
      "causal_effect": 0.35,
      "sample_size": 4200
    },
    {
      "id": 4,
      "type": "leaf",
      "causal_effect": 0.82,
      "sample_size": 3200
    }
  ],
  "leaf_treatment_effects": {
    "1": { "uplift": 0.12, "sample_size": 8500 },
    "3": { "uplift": 0.35, "sample_size": 4200 },
    "4": { "uplift": 0.82, "sample_size": 3200 }
  }

```

大模型接收此 JSON 后，"翻译"为：*"设备更换频次 > 2 且交易金额 > 1000 的用户群体，拦截干预的因果增益为 0.82（样本量 3200），建议对该群体启用拦截策略"*，同时输出对应的规则 DSL。

#### 公平性合规审查

因果决策树可能输出与年龄、性别、地域、民族等法律保护特征高度相关的因果路径。LLM 策略转化层在生成规则 DSL 时，必须执行**公平性合规审查**：

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1. 受保护特征扫描 | 遍历因果树节点，检查拆分特征是否命中受保护特征清单（年龄、性别、地域、民族、宗教等） | 受保护特征清单由合规团队维护 |
| 2. 代理变量检测 | 检查是否存在与受保护特征高度相关的代理变量（如邮编 ≈ 地域、消费品类 ≈ 性别） | 相关性阈值由合规团队设定 |
| 3. 标记与审批 | 命中受保护特征或代理变量的规则自动标记为"需合规审批"，**不可直接进入沙盒跑批** | 合规团队审批通过后方可继续进入三段式漏斗 |

#### 每日风险挖掘

- 因果决策树每日输出新的因果特征和规则建议
- 智能体对输出做有利/有害二次过滤，**仅提取对业务发展有害的因果特征**
- 过滤依据是因果效应权重的方向和强度，而非相关性指标

#### 意图澄清 & 预设 API 组合路由

- 当用户选择"点击探索"深入某条风险时，智能体进行意图澄清
- 通过**预设 API 组合路由**获取数据，**彻底替代 Text-to-SQL**，杜绝大模型直连数据库带来的安全与准确性风险

#### 一键生策 vs 点击探索

| 路径 | 适用场景 | 说明 |
| --- | --- | --- |
| 一键生策 | 已知模式、历史高频出现的同类风险 | 跳过探索，直接基于因果推断结果和历史策略模板生成报告，适合成熟风险的快速响应 |
| 点击探索 | 新出现或复杂的风险信号 | 走完意图澄清 → SOP 排查 → API 数据拉取的完整链路，适合需要深度分析的场景 |

一键生策虽然效率高，但其输出仍须经过完整的三段式验证漏斗（历史推演 → T+1~T+3 空跑 → T+N 结算），不跳过任何质量关卡。

#### 风险报告

报告内容包含：

| 维度 | 说明 |
| --- | --- |
| 风险系数 | 量化风险严重程度 |
| 多维阈值 | 多特征联合调整建议 |
| 业务影响范围 | 受影响用户量级、业务线、预估损失 |
| 管控策略建议 | 可直接落地的策略方案 |

#### 三段式策略验证漏斗

策略从生成到上线，必须经历三个递进阶段，**不可跳级**。每个阶段都有明确的准入条件和退出标准，确保不会因"盲等"或"盲信"引发大面积误杀的生产事故。

##### 阶段一：历史回溯推演 — 数据化业财对抗引擎

**目标**：在不接触线上流量的前提下，通过"策略蓝队 vs 增长红队 + 沙盒客观裁决"的量化对抗，验证策略的拦截效果与业务折损。**纯大模型之间的辩论毫无意义**，真正的对抗必须是"大模型生成规则 → 沙盒跑出绝对真实的业财数据 → 另一个大模型基于业财数据反驳"。

**对抗三方角色定义：**

| 角色 | 身份 | 目标 | 核心约束 |
| --- | --- | --- | --- |
| 策略蓝队 (Risk Agent) | 代表风控部门 | 尽可能多地拦截黑产，提升挽损金额 | 必须输出可执行的规则 DSL |
| 增长红队 (Growth Agent) | 代表业务侧/用户体验 | 保护白名单用户、降低误杀率、保护大盘 GMV | 反驳必须基于沙盒数据，不允许主观臆测 |
| 裁判 (Sandbox Engine) | 冷酷无情的跑批机器 | 只输出客观数据 | 不参与推理，不生成建议 |

**增长红队容忍度：配置中心动态注入（禁止全局写死）**

不同业务线的毛利结构、边际成本与风控容忍度差异极大（例如高客单低毛利品类对 GMV 折损极敏感，而虚拟商品、直播打赏等边际成本接近零的场景对折损容忍度更高；大促与平日的误杀容忍度亦可能不同）。**红队的「底线阈值」不得在 System Prompt 中写死为全局常数。**

| 机制 | 说明 |
| --- | --- |
| 命中维度 | 发起阶段一对抗时，总控路由（Router）或编排层根据当前风险信号命中的**业务线 / 场景 / 活动日历**（如电商线、直播线、大促窗口）解析上下文 |
| 配置来源 | 从**配置中心**读取该维度下的红队容忍度模板（如：白户误杀率上限、GMV 折损相对挽损的比例上限等），支持按业务线、按场景、按活动期覆盖 |
| 注入方式 | 将本轮会话的阈值**动态拼接**进增长红队的 System Prompt 与用户消息，使红队「驳回 / 放行」与当前业务契约一致 |
| 运维职责 | 阈值由业务线与风控共同评审后在配置中心维护；变更留痕，与审计日志对齐 |

> 示例（仅说明形态，非产品默认值）：某直播业务线白户误杀容忍上限配置为 0.8%，某电商线配置为 0.1%；GMV 折损与挽损的对比比例亦可按线配置，而非统一 50%。

**黄金评测集（Golden Dataset）构成：**

由数据仓库定期离线生成，**绝不能靠大模型猜测**：

| 子集 | 来源 | 用途 |
| --- | --- | --- |
| 黑样本集 | 历史 T-30 天已确认的真实欺诈、客诉定性、人工实锤的作弊样本 | 测算拦截率、挽损金额 |
| 绝对白样本集 | 历史 T-30 天的 VIP 用户、高频复购且零违规用户、已完成高级实名的安全用户 | 测算**绝对误杀率**（核心基准） |
| 大盘抽样集 | 随机抽样的自然流量（须包含**新注册用户子集**，覆盖新用户正常行为基线） | 测算大盘水位飙升率 |

> **幸存者偏差声明**：黑样本集仅包含"已发现并确认"的欺诈案例，无法覆盖未被现有策略识别的暗数据（漏网黑产）。因此沙盒评测的拦截率为**保守下限估计**，实际效果可能优于评测结果。评估时须意识到该局限性。

**对抗执行 SOP（4 步闭环）：**

**步骤一：策略提议 (Proposal)**

策略蓝队（Risk Agent）接收因果决策树的结果，生成第一版风控规则 DSL：

```
If 设备更换频次 > 2 AND 交易金额 > 1000 Then 拦截
```

**步骤 1.5：DSL 双层校验 — 语法层 + 语义层**

LLM 生成的 DSL 在进入沙盒前，必须经过**语法校验**与**语义校验**；二者均不通过时打回蓝队，**不计入业务对抗轮次**。仅语法正确而语义背离因果树，会导致沙盒结果失真或浪费对抗轮次，甚至策略方向性错误。

**（1）语法静态校验 (Syntax Linter)**

| 校验项 | 说明 |
| --- | --- |
| 语法完整性 | 括号匹配、关键字拼写、表达式闭合 |
| 字段名合法性 | DSL 中引用的特征字段必须存在于特征字典中 |
| 类型一致性 | 阈值类型与字段类型匹配（如数值字段不允许字符串比较） |

**（2）语义一致性校验 (Semantic Validator，工程侧规则/确定性逻辑)**

在**本轮策略所依据的因果决策树 JSON**（或等价结构化意图）与蓝队输出的 DSL 之间做一致性检查，拦截「语法完美但逻辑反转」类错误：

| 校验项 | 说明 |
| --- | --- |
| 条件方向 | 拆分特征的不等号方向（`>` / `<` / `>=`）是否与因果树节点一致，禁止将「> 2」误写为「< 2」 |
| 组合逻辑 | 因果路径为联合（AND）时，DSL 不得写成宽泛的 OR 导致误放行；根节点到叶节点的特征组合是否与 DSL 条件集合一致（允许工程侧定义可接受的等价变换白名单） |
| 动作与叶节点 | 拦截/放行动作是否与目标叶节点的业务语义一致 |
| 阈值漂移 | 数值阈值是否与因果树 JSON 中叶节点/拆分节点约定一致，超出配置允许的容差则打回 |

- 语义校验**不通过**时，系统向蓝队返回**确定性错误码与差异说明**（例如「特征 X 阈值方向与因果树节点 N 不一致」），要求重写 DSL，**不计入业务对抗轮次**
- 语法与语义**均通过**后，DSL 方可进入沙盒跑批

**步骤二：沙盒客观裁决 (Sandbox Execution)**

规则下发至离线规则引擎，跑批黄金评测集三个子集。沙盒直接吐出**业财成绩单**：

```json
{
  "gmv_saved": 500000,
  "white_user_kill_rate": 0.042,
  "gmv_lost_estimate": 1200000,
  "hit_black_count": 847,
  "hit_white_count": 126,
  "kill_cluster": "大促活跃用户",
  "top3_kill_features": [
    {"feature": "daily_order_count > 5", "contribution": 0.85},
    {"feature": "device_change_freq > 2", "contribution": 0.10},
    {"feature": "night_activity_ratio > 0.7", "contribution": 0.05}
  ]
}
```

**步骤三：红队量化反驳 (Growth Agent Critique)**

增长红队被唤醒时，**本轮容忍度底线已由配置中心在对抗发起时注入**（见上文「增长红队容忍度：配置中心动态注入」），不得在 Prompt 内写死全局常数。红队依据**当前会话的阈值**与沙盒成绩单判断是否驳回。

> *作为业务保护者，你必须依据【本轮注入的 GMV 折损/挽损比例上限、白户误杀率上限等业务底线】与沙盒客观数据对比；当沙盒指标突破任一注入底线时，你必须驳回规则。*

红队基于沙盒成绩单生成带有**具体修改建议**的反驳指令：

| 红队反馈维度 | 内容示例 |
| --- | --- |
| 防守收益 | 命中真实黑样本 847 个，预估挽损 ¥50 万 |
| 业务折损（核心弹药） | 白户误杀率 4.2%，预估 GMV 折损 ¥120 万，远超 ¥50 万挽损 |
| 归因证据 | 误杀白户中 85% 是因为"单日下单次数 > 5"阈值过紧 |
| 修改建议 | 补充"历史客单价分布"特征，或将"设备更换频次"阈值从 2 上调至 4 |

**步骤四：自修正与收敛 (Self-Correction)**

策略蓝队接收红队的"数据 + 建议"，强制修改 DSL 阈值或增加排除条件（白名单逻辑），再次提交沙盒跑批。

**对抗终止条件（满足任一即结束）：**

| 条件 | 说明 |
| --- | --- |
| 沙盒指标落入红队容忍底线内 | 沙盒成绩单满足**本轮从配置中心注入的红队容忍度**（误杀、折损/挽损比等），对抗自然收敛 |
| 达到最大对抗轮次（默认 3 轮） | 系统自动根据**当前会话容忍度配置**下的指标距底线差距，从 3 轮中挑选**综合指标最优版** DSL 作为推荐方案，连同另外两轮的沙盒成绩单一并打包，强制推给**风控总监与业务线总监**进行人工权衡（降低人工研判的认知负荷） |
| 人工仲裁介入 | 风控总监强制裁决，记录裁决理由归档 |

**准出条件**：对抗达成共识（或人工仲裁完成） → 策略人工确认 → 进入阶段二。

##### 阶段二：T+1~T+3 线上空跑预估（水位观察）

**目标**：用较短时间（1~3 天）在真实流量中检测策略是否"发疯"，覆盖工作日与非工作日的周期性波动，实现快速熔断保护。

- 策略下发至规则引擎，设为 **Monitor-Only**（仅观察不拦截），不影响线上用户
- 空跑时长 1~3 天（须至少覆盖一个工作日和一个非工作日，大促期间建议延长至 3 天），由于欺诈标签尚未产出，**不计算 Uplift**，而是比对以下两个水位指标：

| 观测指标 | 正常范围 | 异常判定 |
| --- | --- | --- |
| 新老策略命中重合度 | ≥ 60% | 重合度过低说明新策略逻辑偏移严重 |
| 大盘命中水位飙升比例 | ≤ 原水位的 1.5 倍 | 飙升说明策略可能大面积误杀 |

**异常处理**：

- 水位异常 → **紧急熔断**，自动下线 Monitor 规则，回退策略并告警策略负责人
- 水位正常 → 继续进入阶段三

**准出条件**：水位指标均在正常范围内，且策略负责人确认继续观察。

##### 阶段三：T+N 影子收益结算（最终 Uplift）

**目标**：基于真实欺诈标签，计算策略**如果上线本可以**产生的资金挽损效果和误杀成本，作为策略正式转拦截的最终依据。

> **关键定义对齐**：由于阶段二为 Monitor-Only（仅观察不拦截），这批欺诈交易实际已经发生并造成了损失。因此阶段三结算的是**"影子收益 / 预估挽损"**（如果在空跑期间执行了拦截，本可以挽回多少钱），而非真实已挽回的资金。只有策略正式转为拦截状态后，产生的才是真正的实际挽损。**风控与财务对账时对此极其敏感，指标命名必须严格区分。**

- 等待 T+3 至 T+30 欺诈确认周期结束（具体周期因业务而异）
- 系统拉取携带真实标签（Ground Truth Label）的空跑命中日志，计算：

| 结算指标 | 说明 |
| --- | --- |
| 预估资金挽损（影子收益） | 基于空跑命中日志和真实标签反推：若策略已上线拦截，本可挽回的资金总额 |
| 预估误杀成本 | 空跑期间策略命中但实际为正常用户的数量及对应的客诉/赔付成本预估 |
| Uplift | 对比基线策略的增量效果 |

**标签回传状态机：**

| 阶段 | 策略状态 | 说明 |
| --- | --- | --- |
| 标签未回传 | 观察期 | 策略保持 Monitor 状态，持续收集命中日志 |
| 标签部分回传（< 置信样本量） | 待评估 | 不做最终决策，系统提示"样本量不足，建议延长观察" |
| 标签充分回传 | 可评估 | 计算最终 Uplift，输出策略效果结算报告 |
| 超过最大等待周期仍标签不足 | 超时 | 升级至人工决策，由策略负责人判断是否上线 |

**准出条件**：预估 Uplift 达标且预估误杀成本可接受 → 策略正式转为拦截状态（此后产生的才是真实挽损），归档上线。

##### 上线后持续监控与自动熔断

策略正式转为拦截状态后，并非"一劳永逸"。上线策略必须进入**持续监控期**，防止黑产适应性变化或数据分布漂移导致策略失效或误杀飙升。

| 监控阶段 | 时间窗口 | 观测指标 | 异常处理 |
| --- | --- | --- | --- |
| 短期密集监控 | 上线后 T+1 ~ T+7 | 每日命中量、误杀率、挽损金额 | 任一指标偏离基线 >30% → 自动告警策略负责人 |
| 中期稳定性监控 | 上线后 T+7 ~ T+30 | 周均命中水位、累计误杀率、客诉关联率 | 连续 3 天命中量骤降 >50% 或误杀率突破**该策略绑定业务线在配置中心的告警阈值**（可与阶段一红队容忍度同源或单独配置）→ 自动降级为 Monitor-Only 并告警 |
| 长期衰减监控 | 上线后 T+30 起 | 月度 Uplift 衰减趋势 | Uplift 衰减至基线以下 → 提示策略负责人评估是否下线或迭代 |

**自动熔断机制**：上线策略触发异常阈值时，系统自动将策略从拦截状态回退为 Monitor-Only，同时保留命中日志用于事后分析。策略回退后需重新进入三段式漏斗的阶段一修正后方可再次上线。

## 四、智能体二：案件研判

> 输入 User_ID，自动拉取数据、脱敏清洗、生成研判报告与处置建议。

### 4.1 整体流程

```
输入 User_ID
      │
      ▼
  意图澄清 Clarification
  （聚焦排查时间段与维度）
      │
      ▼
  用户自主选择数据通道：
      ├── [通道A: 预设API拉取] (首选)
      │         │
      │    API异常? → 阻断报错（提示通道B可用），不自动跳转
      │         │
      └── [通道B: 手工上传原始数据] (按需)
                │  强制限制 < 5MB 或 2000行
                │  超限 → 拒绝上传，引导使用代码解释器分析
                │
      ┌─────────┘
      ▼
  复合式隐私脱敏网关 (DLP)
  （字段级 Schema 过滤 + NLP/NER 轻量级小模型隐私识别）
  ⚠ 两条通道的数据均必须经过 DLP 网关
      │
      ▼
  结合 SOP 生成用户研判报告
      │
      ▼
  输出用户处置建议
  （自然语言解释 + User_Disposal_Action 标准化枚举）
```

### 4.2 关键环节说明

#### 意图澄清

- 明确排查的**时间段**和**维度**（交易、登录、设备等），避免无效数据拉取

#### 数据获取：双向独立选择通道

摒弃强制兜底逻辑，用户发起研判时**自主选择**通道：

| 通道 | 定位 | 说明 |
| --- | --- | --- |
| 通道 A：预设 API 拉取（首选） | 输入 User_ID 自动调取 | 标准化接口，结构化返回。若 API 异常/超时，**直接抛出明确错误并阻断**，不自动跳转至上传通道，但在错误提示中附带"如需紧急排查，可使用通道 B 手工上传数据"的替代路径 |
| 通道 B：手工上传（按需） | 满足断网或历史日志回溯场景 | 强制限制文件不超过 **5MB 或 2000 行**。**超限后直接拒绝上传**，系统引导用户使用内置**代码解释器**（沙盒 Python 环境 + Pandas），由大模型编写分析代码处理大文件，而非将原始数据硬塞进 Prompt 上下文（纯文本 Chunking 会导致结构化日志的时间线断裂，丧失时序异常聚集的判断能力） |

#### 复合式隐私脱敏网关（DLP）

摒弃单一且极易漏检的正则表达式脱敏。所有数据必须经过**"字段级 Schema 过滤 + 专用轻量级 NLP/NER 隐私识别小模型"**构成的复合 DLP 网关：

| 脱敏层 | 机制 | 作用 |
| --- | --- | --- |
| 第一层：Schema 过滤 | 根据预设 API 返回的字段 Schema，在结构化层面直接移除已标记的敏感列（手机号、身份证号等） | 处理结构化数据，零漏报 |
| 第二层：NLP/NER 小模型 | 轻量级隐私实体识别模型，扫描非结构化文本中的隐私字段（含异构格式、特殊字符的身份证/银行卡号等） | 处理非结构化数据，覆盖 Schema 遗漏 |

- 两条通道获取的数据**均须经过 DLP 网关**后才能进入大模型上下文
- 脱敏为**高精度不可逆 Masking**，彻底断绝大模型接触明文隐私
- 提供**特征误杀申诉加白机制**：若脱敏导致关键业务特征被误删，策略分析师可提交加白申请，经安全团队审批后将特定字段加入白名单

#### 研判报告生成

- 结合预定义的排查 SOP，将脱敏后的数据组织为结构化报告
- 输出包含用户行为时间线、风险标签、处置建议

#### 处置建议的标准化输出

大模型输出的处置建议**必须同时包含两部分**：自然语言解释（供人工阅读理解）+ 标准化动作枚举指令（供下游管控系统 API 直接消费执行）。

**标准化枚举规范 `User_Disposal_Action`：**

| 动作枚举 | 含义 | 适用场景 |
| --- | --- | --- |
| `BLOCK_DEVICE` | 封禁设备 | 设备指纹异常、模拟器/群控 |
| `FREEZE_FUND` | 冻结资金 | 大额异常交易、疑似洗钱 |
| `FREEZE_ACCOUNT` | 冻结账户 | 账户被盗、严重违规 |
| `DOWNGRADE` | 降级权限 | 信用分下降、轻度违规 |
| `REQUIRE_2FA` | 强制二次验证 | 异地登录、敏感操作 |
| `LIMIT_TRANSACTION` | 限制交易 | 交易频率/金额异常 |
| `MANUAL_REVIEW` | 人工复核 | 风险信号不确定，需人工介入 |
| `NO_ACTION` | 暂不处置 | 研判结论为正常用户 |

**输出格式约束**：

```
{
  "disposal_actions": ["LIMIT_TRANSACTION", "REQUIRE_2FA"],
  "explanation": "用户近7天内3次异地登录且存在2笔大额交易，建议临时限制交易并要求二次验证",
  "evidence_strength": "HIGH",
  "risk_level": "HIGH",
  "evidence_score": 0.82
}
```

> 注：`evidence_score` 宜由研判流水线在结构化输出落库前**计算并写入**（可忽略或覆盖模型给出的数值）；`evidence_strength` 可由模型输出或由规则映射，二者均不得单独作为自动强处置的唯一依据。

- `disposal_actions` 为枚举数组，支持组合下发多个动作
- **`evidence_strength`（证据强度）**：取枚举 `HIGH` / `MEDIUM` / `LOW`，表示研判所依据证据的充分程度；可由大模型按结构化输出，但**不得单独作为唯一分流依据**（见下条）
- **`evidence_score`（客观置信分，推荐由工程侧计算）**：**禁止依赖大模型自报浮点「置信度」**（易产生盲目偏高或随机波动，不宜驱动自动处置）。该分值应由**确定性规则**根据证据链生成，例如：脱敏后仍保留的高危特征命中数、硬实锤规则条数（如多账号同设备、同 IP 批量注册）、时间线异常事件计数、与 SOP 关键检查项的对齐度等，按产品配置的加权公式输出 0~1 分数
- 分流策略：**以 `evidence_score`（若已启用）为主、`evidence_strength` 为辅**；低于配置阈值时强制将 `disposal_actions` 覆盖或追加为 `MANUAL_REVIEW`，仅允许人工确认后下发强处置
- `risk_level` 为风险等级标签，可与下游策略联动
- 下游管控系统（规则引擎/处罚中心）通过 API 直接消费该结构化输出，实现处置闭环

**工程侧强 Schema 校验拦截**：

大模型输出 `disposal_actions` 后，系统必须执行严格的枚举合法性校验：

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1. 合法性校验 | 逐一比对 `User_Disposal_Action` 字典 | 检查每个输出枚举值是否存在于上述标准字典中 |
| 2. 非法值处理 | 自动重试（对用户不可见） | 若发现如 `WARN_USER` 等字典外的幻觉枚举，系统后台自动将校验失败信息回传 LLM 要求修正，最多重试 2 次 |
| 3. 最终兜底 | 强制降级为 `MANUAL_REVIEW` | 若 2 次重试仍包含非法枚举，自动将整条处置建议降级为 `MANUAL_REVIEW`（人工复核），杜绝非法指令下发至处罚中心 |
| 4. 客观置信分闸门 | 与枚举校验独立执行 | 若 `evidence_score` 低于配置阈值，或证据链未满足强处置最低条数，强制 `MANUAL_REVIEW`，不依赖模型自评置信度 |

## 五、智能体三：风控解决方案

> 将模糊的业务需求转化为可落地的风控方案，经专家验收后归档入库。

### 5.1 整体流程

```
模糊业务需求输入
      │
      ▼
  意图澄清 Clarification
  （引导式追问业务体量与作弊细节）
      │
      ▼
  业务场景识别
      │
      ▼
  转化为精准 Skills 标签
  （Risk_Skill_Matrix）
      │
      ▼
  基于 LLM-Wiki 知识网进行上下文构建
  （强制优先准则：现行策略 > 历史方案）
      │
      ▼
  基于专属 SOP 模板的报告生产
  （Report_Agent）
      │
      ▼
  生成业务解决方案草稿
      │
      ▼
  专家人工修改与验收验证
  （Human-in-the-loop）
      │
      ▼ （拒绝投毒，仅验收通过后入库）
  大模型管理员结构化区块增量更新 Wiki 节点（高亮 Diff 供专家审查）
  （依赖底层 Git 进行物理版本控制）
```

### 5.2 关键环节说明

#### 意图澄清

- 不接受模糊输入，通过**引导式追问**主动收集业务体量、作弊手法、历史对抗经验等关键信息
- 确保后续方案有足够的业务上下文支撑

#### Risk_Skill_Matrix

- 将自然语言描述的业务场景转化为**精准的 Skills 标签**
- 标签用于后续 LLM-Wiki 知识网的精确定位

#### LLM-Wiki 知识编译层（替代传统 Vector RAG）

彻底摒弃碎片化且极易丢失上下文的传统向量检索。采用 **LLM-Wiki 模式**，大模型作为知识库的"图书管理员"：

**Wiki 知识网结构：**

| 文档层级 | 内容 | 示例 |
| --- | --- | --- |
| 总览索引页 | 全部场景的分类目录与关联关系 | `index.md` — 列出所有风控场景及跳转链接 |
| 策略概念页 | 每类策略的原理、适用条件、核心参数 | `concepts/device-fingerprint.md` |
| 场景实体页 | 具体场景的完整方案（现行策略 + 历史方案） | `scenes/live-reward/solution.md` |

**知识库查询与绝对优先级冲突解决：**

生成方案草稿时，基于编译整理后的 Wiki 网络构建上下文。在 System Prompt 中强制注入绝对优先级原则：

> *"当检索到的历史优秀方案与现役运行的策略规则发生逻辑或阈值冲突时，必须以【现行策略】为绝对基准，严禁输出破坏现行基线规则的缝合建议。"*

#### 专属 SOP 模板

- **不同场景关注点不同**，必须调用不同的 SOP 分析步骤
- Report_Agent 根据场景标签选择对应模板，保证输出结构的针对性

#### Human-in-the-loop 验收

- 方案草稿交由领域专家修改与验收
- **拒绝投毒**：仅验收通过的内容才会入库，防止低质量或错误方案污染知识库

#### 方案入库与 Wiki 迭代（替代 Diff 碎片检索）

方案经验收通过后，**大模型管理员**对 Wiki 执行**结构化区块增量更新**（而非全文覆盖重写）：

| 项目 | 说明 |
| --- | --- |
| Wiki 结构约束 | 每个场景实体页的 Markdown 必须包含**固定 Header 标签**（如 `## 现行策略拦截线`、`## 白名单豁免条件`、`## 历史特征池`、`## 关联风险场景`）。大模型仅允许对特定 Section 进行增量修改或新增 Section，**禁止覆盖整个文件** |
| 入库方式 | 大模型管理员定位到需修改的 Section，执行区块级增量编辑，保持其余 Section 内容不变，杜绝"灾难性遗忘"（大模型重写全文时可能遗漏隐蔽但关键的兜底规则） |
| 验收 Diff 展示 | 专家 Human-in-the-loop 验收时，系统**必须高亮展示 Diff**（新增了什么、删除了什么、修改了什么），而非仅展示全文，确保审查者能在长文档中快速发现关键变更 |
| 版本控制 | 底层依赖 **Git** 提供面向人类审查的版本历史，支持任意版本回滚 |
| 大模型读取范围 | 生成方案时**永远只阅读最新态的 Wiki 实体文本**，从根本上消除直接检索 Diff 摘要导致的语义幻觉 |
| 冲突防护 | 更新前对比现行策略页，若新方案与现行策略矛盾则拒绝入库并告警 |
| 并发控制 | 同一场景实体页同一时刻仅允许一个方案入库操作（乐观锁机制）。若多个验收通过的方案涉及同一实体页，系统按验收通过时间排队依次执行，后续方案需基于前序方案更新后的最新版本重新做冲突检测 |

## 六、智能体间协作机制

三个智能体并非完全孤立运行，实际业务中存在明确的跨智能体信号流转需求。

### 6.1 协作拓扑

```
  风险挖掘智能体 ──── 批量异常用户清单 ────→ 案件研判智能体
        ▲                                        │
        │                                        │
  效果数据回流                           新作弊模式反馈
  （策略Uplift结果）                              │
        │                                        ▼
  风控解决方案智能体 ←── 案件模式归纳 ──────────────┘
        │
        └── 方案上线后监控数据 ──→ 风险挖掘智能体（纳入因果推断输入）
```

### 6.2 协作场景

| 触发方 | 接收方 | 协作信号 | 触发条件 |
| --- | --- | --- | --- |
| 风险挖掘 | 案件研判 | 批量异常用户 ID 清单 | 风险挖掘发现批量异常，需逐案验证 |
| 案件研判 | 风险挖掘 | 新作弊模式特征 | 研判中发现因果推断未覆盖的新模式 |
| 案件研判 | 解决方案 | 案件模式归纳报告 | 同类案件累积超过阈值，需系统性方案应对 |
| 解决方案 | 风险挖掘 | 方案上线后的监控数据 | 新方案效果数据回流，供下一轮因果推断消费 |

### 6.3 协作原则

- 协作通过**异步消息/任务队列**传递，不要求智能体间实时调用
- 每次跨智能体传递的数据必须经过复合 DLP 网关
- 协作信号仅做"建议触发"，接收方智能体保留独立判断权

## 七、典型用户旅程

### 旅程一：策略分析师的一天（风险挖掘）

> 策略分析师小王每日使用风险挖掘智能体进行风险巡检与策略迭代。

1. **早上 9:00**：小王打开系统，看到昨日因果决策树产出 5 条风险信号（因果效应权重排序），其中 2 条标记为"高危"
2. **9:10**：点击第 1 条高危风险的"点击探索"，系统追问排查维度，小王选择"交易金额 + 设备指纹"
3. **9:15**：系统调用预设 API 拉取数据，展示排查链路 SOP 建议，小王确认 API 调用参数
4. **9:25**：系统展示风险报告（LLM 基于因果决策树 JSON 翻译），包含因果效应权重 0.82、因果路径解释、预估影响 12,000 用户，同时输出第一版规则 DSL
5. **9:30**：小王发起**阶段一业财对抗** — 策略蓝队提交规则 DSL → 沙盒跑批黄金评测集 → 成绩单显示：挽损 ¥50 万、白户误杀率 4.2%、GMV 折损 ¥120 万。增长红队驳回：*"误杀集中在大促活跃用户，85% 因'单日下单 > 5'阈值过紧"*
6. **9:40**：蓝队修正 DSL，上调阈值并补充客单价排除条件。第二轮沙盒跑批：白户误杀率降至 0.3%、挽损 ¥42 万 > 折损 ¥18 万，红队确认达标，对抗收敛
7. **9:45**：小王确认策略，进入**阶段二：T+1~T+3 空跑**，系统下发 Monitor-Only 规则
8. **第 3 天上午**：系统反馈空跑期内水位正常（新老策略命中重合度 78%，大盘水位无飙升，已覆盖工作日与非工作日），进入**阶段三**继续观察
9. **第 5 天**：标签回传充分，**阶段三影子收益结算**显示预估挽损 ¥42 万、预估误杀率 0.4%、Uplift +18%，小王将策略正式转为拦截状态（此后产生真实挽损）并导出上线
10. **9:50**：对第 2 条高危风险点击"一键生策"（该风险为历史高频模式），系统快速生成并展示包含因果效应权重、业务影响范围及规则 DSL 的风险评估报告，随后自动将该策略送入阶段一业财对抗沙盒

### 旅程二：研判人员的排查工作（案件研判）

> 研判人员小李收到可疑用户举报，使用案件研判智能体进行排查。

1. 小李输入 User_ID `U_20260413_7721`，系统追问排查时间段
2. 小李选择"近 7 天"+"交易 + 登录维度"，选择通道 A（预设 API 拉取）
3. 系统调用预设 API 拉取数据，经**复合 DLP 网关**（Schema 过滤 + NER 小模型）清洗后展示
4. 系统结合 SOP 生成研判报告：用户 7 天内 3 次异地登录、2 笔大额交易、设备指纹异常
5. 系统输出处置建议：自然语言为"建议临时限制交易 + 二次验证"，同时输出标准化动作 `["LIMIT_TRANSACTION", "REQUIRE_2FA"]`；工程侧根据高危特征命中与时间线事件计算 `evidence_score` 达标、`evidence_strength` 为 HIGH，小李审核后一键下发至处罚中心执行

### 旅程三：架构师的方案设计（风控解决方案）

> 业务安全架构师老张需要为新上线的直播打赏场景设计风控方案。

1. 老张输入"直播打赏场景需要风控方案"
2. 系统引导式追问：日均打赏流水、主要作弊手法、是否有历史方案
3. 老张补充：日均 500 万、主要是机器人刷量和洗钱、无历史方案
4. 系统识别场景为"直播打赏"，匹配 Skills 标签，从 **LLM-Wiki 知识网**中定位相关场景实体页（2 份同类方案 + 3 份现行策略），以现行策略为绝对基准构建上下文
5. 系统基于"虚拟货币交易 SOP 模板"生成方案草稿
6. 老张修改了用户分层策略部分，验收通过
7. 大模型管理员自动更新 Wiki 中 `scenes/live-reward/solution.md` 实体页，Git 记录版本变更

## 八、角色与权限模型

### 8.1 角色定义

| 角色 | 可访问的智能体 | 核心权限 |

- **脱敏策略修改**：仅安全团队有权调整 DLP 脱敏策略（含 Schema 白名单配置与 NER 模型更新）

## 九、度量体系

### 9.1 北极星指标

**风控策略从发现到上线的端到端周期（天）** — 衡量整体平台对风控效率的提升效果。

### 9.2 各智能体核心指标

| 智能体 | 指标 | 定义 | 目标 |
| --- | --- | --- | --- |
| 风险挖掘 | 策略上线周期 | 从风险信号产出到策略正式转拦截的天数 | ≤ 5 天（当前人工基线约 15 天） |
| 风险挖掘 | 策略 Uplift 中位数 | 所有上线策略的阶段三最终 Uplift 中位值 | ≥ 10% |
| 风险挖掘 | 白户误杀率 | 黄金评测集绝对白样本误杀率（阶段一沙盒）+ 阶段三预估误杀率（影子收益）；**达标线以配置中心按业务线/场景为准**，文档中的全局数值仅作默认基线示例 | 默认基线示例 ≤ 0.5%（各线可配置为更严或更宽） |
| 风险挖掘 | 业财对抗平均轮次 | 阶段一蓝队-红队对抗收敛所需轮次 | ≤ 2 轮 |
| 风险挖掘 | 挽损/折损比 | 沙盒成绩单中 GMV Saved / GMV Lost | ≥ 2:1 |
| 风险挖掘 | 空跑熔断率 | 阶段二空跑（T+1~T+3）因水位异常触发熔断的策略占比 | ≤ 10% |
| 风险挖掘 | 漏斗通过率 | 从阶段一进入到阶段三最终上线的策略比例 | ≥ 60% |
| 案件研判 | 系统处理延迟（System TTR） | 从参数确认完毕到系统输出处置建议的纯处理耗时（API 拉取 + DLP + LLM 推理），**不含意图澄清交互轮次** | ≤ 1 分钟 |
| 案件研判 | 意图澄清耗时 | 从用户首次输入 User_ID 到参数确认完毕的交互耗时（含追问轮次） | ≤ 2 轮追问，系统侧每轮响应 ≤ 5 秒 |
| 案件研判 | 人工决策闭环耗时（Human Decision Time） | 从系统输出处置建议到人工完成阅读、修改并点击下发的总时间 | ≤ 5 分钟（当前基线约 2 小时） |
| 案件研判 | 报告采纳率 | 研判人员接受系统建议（无需大幅修改）的比例 | ≥ 70% |
| 案件研判 | 处置准确率 | 处置建议与最终事实一致的比例 | ≥ 85% |
| 案件研判 | 处置闭环率 | 标准化枚举指令被下游系统成功消费执行的比例 | ≥ 90% |
| 解决方案 | 首次验收通过率 | 方案草稿无需大幅修改直接通过的比例 | ≥ 50% |
| 解决方案 | 知识库复用率 | 新方案中引用已有知识的比例 | ≥ 60% |
| 解决方案 | 方案交付周期 | 从需求输入到方案验收归档的天数 | ≤ 3 天（当前基线约 10 天） |

### 9.3 平台健康度指标

| 指标 | 定义 | 告警阈值 |
| --- | --- | --- |
| 大模型调用成功率 | 请求成功 / 总请求 | < 99% |
| API 平均响应时间 | 预设 API 的 P95 延迟 | > 3s |
| DLP 漏检率 | 抽检中发现的未脱敏 PII 占比（Schema + NER 双层后） | > 0（任何漏检都需修复） |
| Wiki 知识网健康度 | Wiki 实体页总数 / 过期未更新页数 | 过期占比 > 20% 需触发审查 |

## 十、分期规划与 MVP

### 10.1 分期概览

```
Phase 1 (MVP)                Phase 2                    Phase 3
─────────────────          ─────────────────          ─────────────────
案件研判智能体（完整链路）    风险挖掘智能体              风控解决方案智能体
+ 复合 DLP 脱敏网关          + 因果决策树接入              + LLM-Wiki 知识网
+ 预设 API 框架             + 业财对抗沙盒引擎            + 知识库防投毒 + Git 版控
+ 总控路由（单智能体+兜底）   + Uplift 验证               + 智能体间协作
                            + 总控路由（双智能体）         + 总控路由（三智能体）
                                                       + 度量仪表盘
```

### 10.2 MVP 选择理由

| 决策维度 | 案件研判优先的理由 |
| --- | --- |
| 用户痛点紧迫度 | 案件排查是日常高频操作，每天每人 5-10 单，效率提升立竿见影 |
| 技术复杂度 | 链路最短（7 个节点），不依赖因果推断等重离线系统 |
| 价值验证速度 | 2-4 周可交付 MVP，快速收集用户反馈 |
| 基础设施复用 | 复合 DLP 网关和预设 API 框架是其他两个智能体的共用基础 |

### 10.3 各阶段里程碑

| 阶段 | 周期 | 交付物 | 验收标准 |
| --- | --- | --- | --- |

## 十一、异常与降级策略

### 11.1 设计原则

- **不做静默 fallback**：出错时明确告知用户发生了什么、影响范围、预计恢复时间
- **可重试的自动重试，不可重试的快速失败**
- **核心安全机制（脱敏）不允许降级**

### 11.2 异常场景处理

| 异常场景 | 影响范围 | 处理方式 |
| --- | --- | --- |
| 大模型服务不可用 | 报告生成、意图澄清 | 自动重试 3 次（间隔指数退避）；全部失败后提示用户"大模型服务暂不可用，已缓存当前数据，恢复后可继续" |
| 预设 API 超时/报错 | 数据拉取 | 自动重试 2 次；失败后**直接阻断流程并报错**（提示"底层数据接口 XXX 异常，状态码 YYY，已通知研发抢修。如需紧急排查，可使用通道 B 手工上传数据"），**不**自动跳转至手动上传通道，但在错误信息中提示替代路径 |
| 因果决策树输出为空 | 风险挖掘当日无因果特征 | 在仪表盘标注"今日无新增因果风险信号"，不生成空报告 |
| 黄金评测集过期/缺失 | 阶段一对抗无法执行 | **阻断对抗流程**，提示"黄金评测集已过期（超过 30 天），请联系数据团队更新"，不允许用旧数据跑批 |
| DLP 脱敏网关异常 | 数据无法进入大模型 | **不降级，直接阻断流程**，告知用户"DLP 脱敏服务异常，为保障数据安全已暂停处理"，同时告警安全团队 |
| DLP NER 模型漏检 | PII 泄露风险 | 安全团队定期抽检；发现漏检后立即更新 NER 模型和 Schema 规则，并对已处理数据发起回溯审查 |
| 沙盒引擎执行超时/OOM | 阶段一业财对抗 | 沙盒资源按评测集规模弹性配置（设置合理的 CPU/内存/超时上限，具体参数由运维根据黄金评测集实际规模调整）；超限自动终止并提示"评测集数据量过大，建议抽样后重试" |
| T+1~T+3 空跑水位异常 | 阶段二线上空跑 | **紧急熔断**：自动下线 Monitor 规则、回退策略，告警策略负责人；策略回退至阶段一修正后重新进入漏斗 |
| Wiki 知识网 Git 仓库异常 | 方案上下文构建 | 自动检测 Git 仓库完整性，异常时回滚至最近健康 commit；恢复期间提示用户"知识库暂不可用" |
| DSL 语法校验失败 | 蓝队输出的规则 DSL 无法通过 Syntax Linter | 系统自动打回给蓝队重写，**不计入业务对抗轮次**；连续 3 次语法失败则阻断并告警"DSL 生成质量异常" |
| DSL 语义一致性校验失败 | DSL 与因果决策树 JSON 在方向、组合逻辑或阈值上不一致 | 系统返回确定性差异说明并打回蓝队重写，**不计入业务对抗轮次**；连续失败则阻断并告警 |
| 处置枚举 Schema 校验失败 | 大模型输出非法枚举值 | 后台自动重试 2 次要求 LLM 修正（对用户不可见）；仍失败则强制降级为 `MANUAL_REVIEW` |
| 客观置信分未达强处置门槛 | `evidence_score` 低于配置阈值或硬实锤证据不足 | 强制 `MANUAL_REVIEW`，不允许自动下发强处置；不依据模型自报浮点置信度 |
| 手工上传超限 | 文件超过 5MB/2000 行 | **拒绝上传**，引导用户使用内置代码解释器（沙盒 Python + Pandas）分析大文件 |
| 多意图编排失败 | Router 无法解析子任务依赖 | 降级为让用户逐一描述单一意图，最多追问 1 轮 |
| 路由意图识别失败 | 请求无法分发 | 要求用户补充描述，最多追问 2 轮；仍无法识别时提示用户选择目标智能体 |
| 路由意图误分类 | 请求被分发至错误的智能体 | 智能体执行前向用户展示意图确认："我理解您的意图是 XX，即将调用 YY 智能体，是否正确？"；用户否认后回退至 Router 重新分类或手动选择目标智能体 |

## 十二、核心设计原则

### 12.1 大模型边界清晰

| 大模型做什么 | 大模型不做什么 |
| --- | --- |
| 翻译因果决策树 JSON 为归因报告和规则 DSL | 因果推断计算（由 EconML / CausalML 完成） |
| 作为策略蓝队/增长红队进行业财对抗辩论 | 猜测误杀率（由沙盒跑批黄金评测集客观裁决） |
| 意图理解与澄清 | 直接查询数据库（Text-to-SQL） |
| 报告生成与 SOP 编排 | 接触明文隐私数据 |
| 知识召回与融合 | 独立做出上线决策 |
| Wiki 区块级增量编辑 | 全文覆盖重写 Wiki 实体页（防灾难性遗忘） |
| 生成规则 DSL | DSL 语法与语义一致性校验（由 Syntax Linter 与 Semantic Validator 完成） |

### 12.2 Human-in-the-loop 贯穿始终

- **风险挖掘**：阶段一人工确认 → 阶段二水位熔断决策 → 阶段三结算后转拦截审批
- **案件研判**：研判报告供人工审核；标准化处置指令经**枚举 Schema 校验 + 客观置信分闸门**后需人工确认方可下发强处置
- **解决方案**：专家验收是入库的必要条件，验收时必须高亮 Diff 展示 Wiki 变更内容

### 12.3 数据安全优先

- 隐私数据**永不离开复合 DLP 脱敏网关**
- 预设 API 替代自由查询，收窄数据暴露面
- 知识入库需人工验收，Wiki 更新前自动对比现行策略防止冲突

## 十三、关键技术决策

| 决策项 | 选择 | 理由 |
| --- | --- | --- |
| 因果推断 | 因果决策树（EconML / CausalML），算法出树 → LLM 出策略 | 算法保证统计严谨性，大模型负责翻译为可读规则，各司其职 |
| 数据查询 | 预设 API 组合路由 | 安全性、可审计性远优于 Text-to-SQL |
| 隐私保护 | 复合式 DLP 脱敏网关（Schema + NLP/NER 模型） | 替代低效正则，双层拦截覆盖结构化与非结构化数据 |
| DSL 质量保障 | 语法静态校验（Syntax Linter）+ 语义一致性校验（Semantic Validator） | 语法与语义均通过后再入沙盒；防止「语法正确但逻辑反转」导致误判或浪费对抗轮次 |
| 红队容忍度 | 配置中心按业务线/场景/活动期维护，对抗发起时动态注入增长红队 Prompt | 避免全局写死底线导致业务不认或风控失效；与毛利结构、边际成本差异对齐 |
| 对抗兜底 | 3 轮未收敛时推送综合指标最优版 | 避免将"震荡最差版"推给人工，降低研判认知负荷；排序依据为**当次会话注入的容忍度配置**下的综合指标 |
| 策略验证 | 三段式漏斗：阶段一数据化业财对抗（蓝队/红队/沙盒）→ 阶段二 T+1~T+3 空跑 → 阶段三 T+N 影子收益结算 | 递进式验证杜绝"盲等盲信"；对抗基于黄金评测集客观裁决，不做纯 LLM 辩论 |
| Uplift 计算 | 基于真实标签的影子收益（阶段三结算） | 空跑期间计算预估挽损，转拦截后才是实际挽损，指标命名严格区分 |
| 知识管理 | LLM-Wiki 架构 + 结构化区块增量更新 | 大模型以 Section 为粒度增量修改 Wiki（禁止全文覆盖），验收时高亮 Diff，杜绝灾难性遗忘 |
| 处置输出保障 | User_Disposal_Action 强 Schema 校验 + 客观证据分（evidence_score）闸门 | 禁止以模型自报浮点置信度驱动分流；枚举合法且证据分达标方可自动链路，否则 MANUAL_REVIEW |
| 大文件分析 | 代码解释器（沙盒 Python + Pandas） | 超限文件拒绝直接上传，引导 LLM 编写代码分析，避免纯文本 Chunking 导致时序断裂 |
| 多意图处理 | Router 多意图解析与任务编排 | 混合 Prompt 自动拆解并按依赖有序调度，支持跨智能体串行/并行 |
| SOP 模板 | 场景化分模板 | 不同风险场景的分析维度差异大，统一模板无法覆盖 |

## 十四、安全与合规

### 14.1 隐私保护

- **复合 DLP 网关**（Schema 过滤 + NLP/NER 小模型）对手机号、身份证号、银行卡号等做**高精度不可逆 Masking**
- 脱敏在本地完成，**大模型上下文中不存在任何明文隐私数据**
- 提供特征误杀申诉加白机制，平衡安全性与业务可用性

### 14.2 数据访问控制

- 所有数据获取通过预设 API，不开放自由查询接口
- API 调用日志可审计、可追溯

### 14.3 知识库防投毒

- 仅经过专家验收的内容可入库
- 入库时大模型管理员直接更新 Wiki 实体页，底层 Git 提供完整版本历史，支持回滚
- 更新前自动对比现行策略页，若新方案与现行策略矛盾则拒绝入库并告警

### 14.4 审计日志规范

所有关键操作必须留存审计日志，满足合规审计与事后追溯需求。

**日志字段规范：**

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| `operator_id` | 操作人 ID | `U_analyst_001` |
| `operator_role` | 操作人角色 | `ANALYST` / `DIRECTOR` |
| `action` | 操作类型 | `CREATE_CASE` / `APPROVE_STRATEGY` / `UPDATE_WIKI` / `DISPATCH_DISPOSAL` |
| `target_type` | 被操作对象类型 | `CASE` / `STRATEGY` / `WIKI_PAGE` / `GOLDEN_DATASET` |
| `target_id` | 被操作对象 ID | `STR_20260413_001` |
| `detail` | 操作详情（结构化 JSON） | 含操作前后状态、修改内容摘要 |
| `timestamp` | 操作时间（UTC） | `2026-04-13T09:30:00Z` |
| `result` | 操作结果 | `SUCCESS` / `FAILED` / `REJECTED` |

**留存策略：**

- 审计日志保留 **3 年**（满足金融风控合规审计周期要求）
- 日志存储不可篡改（仅允许追加写入，禁止修改和删除）
- 仅团队负责人和安全团队有权查阅完整审计日志；其他角色仅可查阅自己的操作记录

- 所有外部 API 调用必须经过**统一网关**，记录调用日志
- 数据从外部系统进入本平台后，必须经过复合 DLP 网关之后才能进入大模型上下文
- 因果推断服务与大模型服务可独立部署，互不依赖
- **红队容忍度、阶段一/上线后误杀相关告警阈值**等与业务线绑定的参数以配置中心为唯一来源，禁止在代码或 Prompt 模板中写死全局常数

## 十六、差异化优势

### 与传统风控工具对比

| 维度 | 传统规则引擎 + BI 看板 | 风控 SaaS（同盾、邦盛等） | 本产品 |
| --- | --- | --- | --- |
| 风险发现 | 依赖人工配置规则（相关性），无自动挖掘 | 提供通用模型，但不贴合业务 | 因果决策树剔除虚假相关 + LLM 翻译为可读规则 |
| 策略验证 | 人工 Excel 计算 | 部分提供 A/B 测试 | 数据化业财对抗引擎（蓝队/红队/沙盒裁判）+ 三段式漏斗 |
| 案件排查 | 多系统手动查询拼接 | 提供标准化报告 | 一键拉取 + 脱敏 + SOP 结构化报告 |
| 方案沉淀 | 散落在文档中 | 不涉及 | LLM-Wiki 知识网 + 防投毒验收 + Git 版本控制 |
| 数据安全 | 依赖 DBA 权限管理 | 数据上传至第三方 | 本地部署 + 复合 DLP 脱敏 + 预设 API（数据不出域） |
| 大模型可控性 | 不涉及 | 部分集成 LLM，边界不清 | 明确大模型边界，不做因果推断、不做 Text-to-SQL、不接触 PII |

### 核心差异化定位

**"自有数据不出域 + 大模型边界清晰 + 策略全链路闭环"** — 这是本产品区别于通用 SaaS 和传统工具的根本定位。通用 SaaS 需要数据上传至第三方，传统工具缺乏智能化能力，本产品在保证数据主权的前提下实现风控全流程智能化。

## 十七、名词表

| 术语 | 含义 |
| --- | --- |
| 总控路由 (Router) | 顶层意图分发模块，将用户请求路由至对应智能体 |
| 因果决策树 | 基于 Uplift Modeling 的因果森林算法输出，剔除虚假相关后保留真正因果特征的决策树结构 |
| EconML / CausalML | 微软/Uber 开源的因果推断算法库，用于生成因果决策树 JSON |
| 规则 DSL | 策略的结构化表达（如 `If X > N Then 拦截`），可直接下发规则引擎执行 |
| 策略蓝队 (Risk Agent) | 代表风控部门的大模型智能体，目标是最大化拦截和挽损 |
| 增长红队 (Growth Agent) | 代表业务侧的大模型智能体，目标是保护白户和大盘 GMV，基于沙盒数据进行量化反驳；**驳回底线由配置中心按业务线/场景动态注入，非全局写死** |
| 沙盒引擎 (Sandbox Engine) | 冷酷无情的跑批裁判，只输出客观业财数据，不参与推理 |
| 黄金评测集 (Golden Dataset) | 由数据仓库离线生成的标准评测数据，含黑样本集、绝对白样本集、大盘抽样集三个子集 |
| 业财成绩单 | 沙盒跑批后输出的量化指标矩阵，含挽损金额、白户误杀率、GMV 折损预估、误杀归因 |
| 预设 API 组合路由 | 预先定义的 API 调用组合，替代 Text-to-SQL 实现安全的数据查询 |
| 复合 DLP 网关 | Data Loss Prevention，结合字段级 Schema 过滤与轻量级 NLP/NER 隐私识别小模型的双层隐私拦截系统，替代低效正则脱敏 |
| LLM-Wiki 模式 | 由大模型作为"图书管理员"持续维护的结构化 Markdown 知识网络（含总览索引页、策略概念页、场景实体页），底层依赖 Git 版本控制，替代碎片化的向量 RAG + Diff 检索 |
| Risk_Skill_Matrix | 业务场景到 Skills 标签的映射矩阵 |
| Report_Agent | 基于 SOP 模板生成结构化报告的子智能体 |
| DSL 语法静态校验 (Syntax Linter) | LLM 输出 DSL 后、进入沙盒前的工程侧语法校验层，语法错误直接打回不计入对抗轮次 |
| DSL 语义一致性校验 (Semantic Validator) | 将 DSL 与本轮因果决策树 JSON 做确定性一致性检查（条件方向、AND/OR、阈值容差等），防止逻辑反转；不通过则打回不计入对抗轮次 |
| 红队容忍度配置 | 配置中心中按业务线/场景/活动期维护的白户误杀上限、GMV 折损与挽损比例上限等；对抗发起时注入增长红队 Prompt |
| evidence_strength | 证据强度枚举（HIGH/MEDIUM/LOW），可结构化输出，但分流须与客观 evidence_score 联用 |
| evidence_score | 客观置信分（0~1），由工程侧依据高危特征命中、硬实锤条数、时间线异常等确定性规则加权计算，**禁止用模型自报浮点置信度替代** |
| 三段式验证漏斗 | 策略上线前必经的三阶段递进验证：数据化业财对抗 → T+1~T+3 空跑水位观察 → T+N 影子收益结算 |
| T+1~T+3 水位观察 | 策略以 Monitor-Only 模式空跑 1~3 天（须覆盖工作日与非工作日），通过命中重合度和大盘水位飙升比例判断策略是否异常 |
| 影子收益 / 预估挽损 | 空跑期间策略未真正拦截，基于命中日志和真实标签反推"若已上线本可挽回"的资金，仅供评估决策，非实际财务入账 |
| Uplift 计算 | 基于真实黑白标签衡量策略相比基线带来的增量效果（三段式漏斗阶段三） |
| User_Disposal_Action | 标准化处置动作枚举字典，定义案件研判输出的机器可读处置指令，工程侧强制 Schema 校验确保合法性 |
| 代码解释器 (Code Interpreter) | 沙盒 Python 环境 + Pandas，用于分析超限大文件（替代纯文本 Chunking），避免时序断裂 |
| 多意图编排 | Router 对混合意图 Prompt 的拆解、依赖分析与有序调度能力，支持跨智能体串行/并行编排 |
| Human-in-the-loop | 关键决策节点保留人工审核，不完全依赖自动化 |
| Monitor 规则空跑 | 策略以监控模式运行，仅记录不拦截，用于离线评估 |
| SOP | 标准作业流程，定义每类场景的分析步骤与输出规范 |
