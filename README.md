# 智能风控助手 — 大模型 + 多智能体企业级落地方案

用大模型做企业系统，核心挑战不是调 API，而是：**大模型的输出是概率性的，但业务决策需要确定性**。

本项目以风控场景为载体，完整展示一套"大模型只做翻译和建议、关键判断用确定性代码保障"的架构方案——从产品设计到技术方案到可运行代码到面试问答，全链路打通。

---

## 为什么做这个项目

市面上大模型的落地资料，要么是"调 API 写个聊天机器人"（太玩具，离生产差十万八千里），要么是"看论文复现模型"（太学术，解决不了工程问题）。

真正把大模型塞进企业系统时，你会撞上一堆教程里不会讲的问题：

- 大模型生成了一条规则，怎么**证明**它有效，而不是"看起来有道理"？
- 大模型说"这个用户是黑产"，你敢直接拦截吗？误杀了谁负责？
- 多个分析师同时让大模型更新知识库，文件冲突了怎么办？
- 算法训练要跑 2 小时，用户在网页上干等着吗？

这些问题没有哪个单一技术能回答，需要一整套架构设计。本项目就是这套设计的完整记录——不仅写了"怎么做"，还写了"为什么这么做"和"不这么做会怎样"。

## 快速开始

```bash
# 1. 启动数据库和缓存（需要先安装 Docker）
docker compose -f docker-compose.dev.yml up -d

# 2. 安装 Python 依赖
python -m venv .venv && .venv\Scripts\activate  # Windows
pip install -r requirements.txt

# 3. 建表 + 种子数据 + 启动 API
psql -h localhost -U risk_user -d risk_control -f scripts/init_db.sql
uvicorn api.main:app --reload
```

打开 `http://localhost:8000/docs` 看到 Swagger UI 就算成功。详细步骤见 [开发环境搭建.md](开发环境搭建.md)。

## 系统架构（30 秒版）

```
用户（风控分析师）
   │
   ▼
┌─────────────────────────────────────────────────┐
│  FastAPI 网关层                                   │
│  意图识别 → 总控路由 → 分发到三个智能体             │
└────────┬───────────────┬───────────────┬─────────┘
         │               │               │
    ┌────▼────┐    ┌─────▼─────┐   ┌─────▼──────┐
    │风险挖掘  │    │ 案件研判   │   │ 风控解决方案 │
    │智能体    │    │ 智能体     │   │ 智能体      │
    └────┬────┘    └─────┬─────┘   └─────┬──────┘
         │               │               │
         ▼               ▼               ▼
┌─────────────────────────────────────────────────┐
│  共享基础设施                                      │
│  配置中心 │ DLP 脱敏 │ 审计日志 │ LLM API │ Wiki   │
└─────────────────────────────────────────────────┘
         │                                │
    ┌────▼─────────┐              ┌───────▼──────┐
    │ PostgreSQL   │              │ Redis        │
    │ (持久化存储)  │              │ (缓存+队列)  │
    └──────────────┘              └──────────────┘
```

核心设计原则：**大模型是翻译员，不是决策者**。所有关键判断（误杀率、挽损金额、策略上下线）都用确定性代码计算，大模型只负责理解意图、生成报告、翻译 DSL。

## 不做风控？这套架构也能用

本项目的价值不在于"风控"，而在于一套**可迁移的架构模式**——解决"大模型输出不确定但业务需要确定性"这个通用问题。

### 具体场景示例

**智能营销系统**

- 挖掘智能体 → **用户洞察智能体**：从数据中发现高价值人群
- 案件研判 → **投放分析智能体**：评估某个投放策略的 ROI
- 风控解决方案 → **营销策略智能体**：生成投放方案 + 预算分配
- 沙盒评测 → 在历史数据上回测投放策略的预期效果
- 大模型边界不变：大模型写文案、分析报告；预算计算、ROI 预估用代码

**智能客服系统**

- 三个智能体 → **意图识别 / 工单处理 / 知识沉淀**
- 沙盒评测 → 新的自动回复模板在历史工单数据集上测满意度
- Wiki 知识库 → 每次人工客服修正 AI 回答，自动更新 FAQ 库
- 三段式漏斗 → 新话术灰度投放给 5% 用户 → 观察满意度 → 全量

**内容审核系统**

- 三个智能体 → **违规发现 / 案件审查 / 审核策略沉淀**
- 几乎和风控同构：发现可疑内容 → 人工复审确认 → 规则沉淀 → 沙盒评测准确率
- 大模型做内容理解和分类建议，最终审核决策由规则引擎 + 人工完成

**DevOps 智能运维**

- 三个智能体 → **异常检测 / 根因分析 / 自动修复方案**
- 沙盒评测 → 修复脚本在模拟环境中验证，通过后才在生产执行
- Git Wiki → 故障手册自动积累，每次排障后更新

**信贷审批系统**

- 因果推断 → 评估"给这个客户降息是否能提升还款率"（不是"这个客户会不会违约"）
- 分级候选 → 数据足的产品用评分卡模型，新产品冷启动用专家规则 + LLM 辅助
- 三段式漏斗 → 新授信策略先在历史样本回测 → 小规模放量 → 全量

### 迁移时改什么、不改什么


| 不需要改的（通用骨架）                          | 需要替换的（业务特定）             |
| ------------------------------------ | ----------------------- |
| FastAPI + Pydantic + SQLAlchemy 基础设施 | 智能体的具体 Prompt 和业务逻辑     |
| Celery 异步任务框架                        | 任务类型（训练→你的业务计算）         |
| 级联配置模块                               | 配置项内容（阈值→你的业务参数）        |
| Git Wiki 读写框架                        | Wiki 的目录结构和 Markdown 模板 |
| 三段式验证漏斗骨架                            | 每个阶段的具体评测指标             |
| DSL 三层校验框架                           | DSL 的具体语法和语义规则          |
| 审计日志 + DLP 脱敏                        | 脱敏的具体字段和规则              |
